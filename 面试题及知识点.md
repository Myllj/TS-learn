# 面试题常考知识点!!!

## 1、interface 与 type 的异同

`interface`命令与`type`命令作用类似，都可以表示对象类型。

很多对象类型既可以用 interface 表示，也可以用 type 表示。而且，两者往往可以换用，几乎所有的 interface 命令都可以改写为 type 命令。

它们的相似之处，首先表现在都能为对象类型起名。

```typescript
type Country = {
  name: string;
  capital: string;
}

interface Country {
  name: string;
  capital: string;
}
```

上面示例是`type`命令和`interface`命令，分别定义同一个类型。

`class`命令也有类似作用，通过定义一个类，同时定义一个对象类型。但是，它会创造一个值，编译后依然存在。如果只是单纯想要一个类型，应该使用`type`或`interface`。

interface 与 type 的区别有下面几点。

（1）`type`能够表示非对象类型，而`interface`只能表示对象类型（包括数组、函数等）。

（2）`interface`可以继承其他类型，`type`不支持继承。

继承的主要作用是添加属性，`type`定义的对象类型如果想要添加属性，只能使用`&`运算符，重新定义一个类型。

```typescript
type Animal = {
  name: string
}

type Bear = Animal & {
  honey: boolean
}
```

上面示例中，类型`Bear`在`Animal`的基础上添加了一个属性`honey`。

上例的`&`运算符，表示同时具备两个类型的特征，因此可以起到两个对象类型合并的作用。

作为比较，`interface`添加属性，采用的是继承的写法。

```typescript
interface Animal {
  name: string
}

interface Bear extends Animal {
  honey: boolean
}
```

继承时，type 和 interface 是可以换用的。interface 可以继承 type。

```typescript
type Foo = { x: number; };

interface Bar extends Foo {
  y: number;
}
```

type 也可以继承 interface。

```typescript
interface Foo {
  x: number;
}

type Bar = Foo & { y: number; };
```



（3）同名`interface`会自动合并，同名`type`则会报错。也就是说，TypeScript 不允许使用`type`多次定义同一个类型。

```typescript
type A = { foo:number }; // 报错
type A = { bar:number }; // 报错
```

上面示例中，`type`两次定义了类型`A`，导致两行都会报错。

作为比较，`interface`则会自动合并。

```typescript
interface A { foo:number };
interface A { bar:number };

const obj:A = {
  foo: 1,
  bar: 1
};
```

上面示例中，`interface`把类型`A`的两个定义合并在一起。

这表明，interface 是开放的，可以添加属性，type 是封闭的，不能添加属性，只能定义新的 type。



（4）`interface`不能包含属性映射（mapping），`type`可以，详见《映射》一章。

```typescript
interface Point {
  x: number;
  y: number;
}

// 正确
type PointCopy1 = {
  [Key in keyof Point]: Point[Key];
};

// 报错
interface PointCopy2 {
  [Key in keyof Point]: Point[Key];
};
```



（5）`this`关键字只能用于`interface`。

```typescript
// 正确
interface Foo {
  add(num:number): this;
};

// 报错
type Foo = {
  add(num:number): this;
};
```

上面示例中，type 命令声明的方法`add()`，返回`this`就报错了。interface 命令没有这个问题。

下面是返回`this`的实际对象的例子。

```typescript
class Calculator implements Foo {
  result = 0;
  add(num:number) {
    this.result += num;
    return this;
  }
}
```



（6）type 可以扩展原始数据类型，interface 不行。

```typescript
// 正确
type MyStr = string & {
  type: 'new'
};

// 报错
interface MyStr extends string {
  type: 'new'
}
```

上面示例中，type 可以扩展原始数据类型 string，interface 就不行。



（7）`interface`无法表达某些复杂类型（比如交叉类型和联合类型），但是`type`可以。

```typescript
type A = { /* ... */ };
type B = { /* ... */ };

type AorB = A | B;
type AorBwithName = AorB & {
  name: string
};
```

上面示例中，类型`AorB`是一个联合类型，`AorBwithName`则是为`AorB`添加一个属性。这两种运算，`interface`都没法表达。

综上所述，如果有复杂的类型运算，那么没有其他选择只能使用`type`；一般情况下，`interface`灵活性比较高，便于扩充类型或自动合并，建议优先使用。



# （二）、📚 TypeScript 中定义对象的方式全解析

在 TypeScript 中，定义对象类型主要有 4 种核心方式，下面我会对比它们的**语法、异同、适用场景**，并给出日常开发的推荐用法。

------

## 一、4 种核心定义方式

### 1. 匿名对象类型（直接字面量）

直接在变量后声明对象结构，适合临时使用的简单对象。

```typescript
// 语法
let user: { name: string; age: number; isAdmin?: boolean } = {
  name: "张三",
  age: 25,
};
```



### 2. `type` 类型别名

用 `type` 关键字给对象类型起别名，可用于基础类型、联合类型、对象类型。

```typescript
// 语法
type User = {
  name: string;
  age: number;
  isAdmin?: boolean;
};

let user: User = { name: "张三", age: 25 };
```



### 3. `interface` 接口

专门用于定义对象 / 类的结构，支持声明合并和继承，是面向对象场景的首选。

```typescript
// 语法
interface User {
  name: string;
  age: number;
  isAdmin?: boolean;
}

let user: User = { name: "张三", age: 25 };
```



### 4. `class` 类

用类来定义对象，兼具类型定义和实例创建能力，适合需要实例化、有方法的对象。

```typescript
// 语法
class User {
  name: string;
  age: number;
  isAdmin?: boolean;

  constructor(name: string, age: number) {
    this.name = name;
    this.age = age;
  }

  sayHello() {
    console.log(`我是${this.name}`);
  }
}

let user: User = new User("张三", 25);
```

------

## 二、核心异同对比表

| 特性                     | 匿名对象     | `type` 类型别名               | `interface` 接口    | `class` 类               |
| ------------------------ | ------------ | ----------------------------- | ------------------- | ------------------------ |
| **是否支持声明合并**     | ❌            | ❌                             | ✅                   | ❌                        |
| **是否支持继承 / 扩展**  | ❌            | ✅（通过交叉类型 `&`）         | ✅（通过 `extends`） | ✅（通过 `extends`）      |
| **是否能用于非对象类型** | ❌            | ✅（支持基础类型、联合类型等） | ❌                   | ❌                        |
| **是否能被实例化**       | ❌            | ❌                             | ❌                   | ✅                        |
| **是否能添加方法**       | ✅            | ✅                             | ✅                   | ✅（且支持实现）          |
| **是否会编译为 JS**      | ❌            | ❌                             | ❌                   | ✅（会生成构造函数）      |
| **适用场景**             | 临时简单对象 | 所有类型定义、复杂类型组合    | 面向对象、API 契约  | 需要实例化、有方法的对象 |

------

## 三、关键差异详解

### 1. `type` vs `interface`（最常被问到的区别）

#### 相同点

- 都能定义对象类型
- 都支持可选属性、只读属性、函数类型
- 都可以被类实现（`implements`）

#### 核心不同点

| 场景                                        | `interface`              | `type`                     |
| ------------------------------------------- | ------------------------ | -------------------------- |
| **声明合并**                                | 支持：同名接口会自动合并 | 不支持：同名 `type` 会报错 |
| 示例                                        | ```ts                    |                            |
| interface User { name: string }             |                          |                            |
| interface User { age: number }              |                          |                            |
| // 合并后 User 同时有 name 和 age           |                          |                            |
| ```                                         | ```ts                    |                            |
| type User = { name: string }                |                          |                            |
| type User = {age: number} // 报错：重复定义 |                          |                            |



```
| **扩展方式** | 用 `extends` 关键字，更直观 | 用交叉类型 `&`，可组合多个类型 |
| 示例 | ```ts
interface Admin extends User {
  role: string
}
​``` | ```ts
type Admin = User & { role: string }
​``` |
| **适用范围** | 只能定义对象/类结构 | 可定义任意类型（包括联合、元组、基础类型） |

---

### 2. `class` 与其他方式的区别
- **核心差异**：`class` 会被编译为 JavaScript 代码，而 `type`/`interface` 仅在编译阶段存在，运行时会被擦除。
- 优势：能创建实例、实现方法、支持继承和多态，适合业务逻辑中的实体对象。
- 劣势：如果仅用于类型检查，会增加编译后的代码体积，且无法用于简单的类型声明。

---

## 四、日常开发使用建议（按场景选择）
### 1. 优先用 `interface` 的场景
- 定义**公共 API 契约**（如前后端接口数据格式）
- 定义类需要实现的结构（`implements`）
- 需要声明合并（比如扩展第三方库的类型）
- 团队协作的基础类型定义，可读性更强

示例：
​```typescript
// 前后端接口契约
interface UserInfo {
  id: number;
  name: string;
  email: string;
}

// 类实现接口
class UserService implements UserInfo {
  id: number;
  name: string;
  email: string;
  constructor(id: number, name: string, email: string) {
    this.id = id;
    this.name = name;
    this.email = email;
  }
}
```

------



### 2. 优先用 `type` 的场景

- 需要定义**非对象类型**（如联合类型、元组、函数类型）
- 复杂类型组合（交叉类型、映射类型）
- 临时的、一次性的对象类型（无需扩展）

示例：

```typescript
// 联合类型
type Status = "success" | "error" | "loading";

// 函数类型
type Handler = (data: string) => void;

// 交叉类型组合
type AdminUser = User & { role: "admin" | "superAdmin" };
```

------



### 3. 优先用 `class` 的场景

- 需要创建多个实例的对象（如业务实体、工具类）
- 对象需要包含方法、构造函数、继承逻辑
- 需要运行时的类型检查（`instanceof`）

示例：

```typescript
class Order {
  orderId: string;
  amount: number;
  constructor(orderId: string, amount: number) {
    this.orderId = orderId;
    this.amount = amount;
  }

  pay() {
    console.log(`订单 ${this.orderId} 支付 ${this.amount} 元`);
  }
}

const order = new Order("123", 99);
order.pay();
```

------



### 4. 匿名对象类型（仅临时使用）

仅在以下情况使用：

- 函数参数 / 返回值的临时对象类型，无需复用
- 简单的一次性变量声明，无扩展需求

示例：

```typescript
// 函数参数临时对象
function printUser(user: { name: string; age: number }) {
  console.log(`${user.name} is ${user.age} years old`);
}
```

------

## 五、最终推荐总结

1. 对象类型定义：优先 `interface`，需要复杂组合 / 非对象类型时用 `type`

   - 90% 的场景下，`interface` 更符合对象定义的语义，且支持声明合并，扩展性更好。

   

2. 业务实体 / 带方法的对象：用 `class`

   - 兼顾类型定义和运行时能力，适合需要实例化的场景。

   

3. 临时对象 / 一次性使用：用匿名对象类型

   - 无需单独声明类型，简单快捷。

   

------

💡 补充小技巧：

- 不要混用 `interface` 和 `type` 定义对象类型，团队内统一风格即可。
- 对于第三方库扩展，优先用 `interface` 声明合并，避免修改库源码。
- 仅做类型检查时，`interface`/`type` 都可以，`class` 会增加运行时代码，非必要不使用。

## 六、企业级全栈开发：场景化最佳实践与铁律规范

### 1. 前后端同构契约层（shared/types）

**铁律：统一用 interface 定义，绝对不能混用 type/class**

- 目录规范：单独建立`@shared/types`目录，前后端共用，所有 API 的请求 / 响应、基础实体契约全部在这里定义，保证前后端类型 100% 一致，从根源上避免联调 bug。

- 示例：

  ```typescript
  // @shared/types/user.ts
  export interface UserBaseInfo {
    id: number;
    username: string;
    email: string;
    createTime: string;
  }
  
  export interface UserListRequest {
    page: number;
    pageSize: number;
    keyword?: string;
  }
  
  export interface UserListResponse {
    list: UserBaseInfo[];
    total: number;
  }
  ```

  

- 进阶技巧：用 Swagger/OpenAPI 自动生成该目录的 interface，避免手动编写导致的类型不一致，全栈开发提效 50% 以上。

### 2. 后端服务层（NestJS/Express）

#### （1）DTO/VO 层：统一用 class，配合 class-validator

**铁律：后端接收前端请求的 DTO，绝对不能用 interface，必须用 class 做运行时校验**，这是后端数据安全的第一道防线。

#### （2）服务层 / 仓储层：interface 定义抽象，class 实现

**铁律：面向接口编程，先定义 interface 规范，再用 class 实现**，符合依赖倒置原则，方便后续单元测试、服务替换。

#### （3）ORM 实体层：统一用 class，配合装饰器

**铁律：数据库实体必须用 class**，适配 TypeORM/Prisma 的装饰器生态，同时实现类型定义与数据库映射的统一。

### 3. 前端应用层（React/Vue）

#### （1）组件 Props/State：优先用 type

**铁律：组件 Props 用 type 定义**，方便配合`Pick`/`Omit`等工具类型，基于后端契约裁剪类型，保证前后端类型一致，同时适配联合类型等灵活场景。

#### （2）状态管理（Pinia/Redux）：统一用 type

**铁律：State/Action 类型用 type 定义**，适配联合类型、复杂状态组合的场景，类型边界更清晰。

#### （3）通用工具函数 / 请求封装：interface 定义请求 / 响应契约，type 定义工具类型

**铁律：API 请求的入参 / 出参必须复用后端的 interface 契约**，绝对不能单独定义，避免类型不一致。
